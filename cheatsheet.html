L01 — Big Data Foundations & Storage

Big Data — Definitions

Term Definition
Big Data Huge and complex datasets that traditional data processing tools find difficult to process. Not just about size — also about complexity and speed of generation/processing.
Data Information that can be stored and used by a computer program.

The 5 V's of Big Data

Term Definition
Volume Massive size of data (TB → PB → EB).
Velocity Speed at which data is generated and processed.
Variety Different types of data: structured, semi-structured, unstructured.
Veracity Trustworthiness / accuracy of data.
Value Insights and business value extracted from data.

Data Types

Term Definition
Structured Data Organized in rows/columns with a strict schema (relational tables). E.g., bank records.
Semi-structured Data Tagged or labeled but doesn't fit a rigid schema. E.g., JSON, XML.
Unstructured Data No predefined structure. E.g., images, videos, free text, audio.

Schema Approaches

Term Definition
Schema-on-Write Schema defined before data is stored; data validated/transformed at ingestion. Used in DW & traditional BI.
Schema-on-Read Data stored as-is; schema applied at query time. Used in Data Lakes / Big Data.
ETL Extract → Transform → Load. Used with Schema-on-Write (warehouse).
ELT Extract → Load → Transform. Used with Schema-on-Read (lake — load raw, transform at read).

Big Data Ecosystem (5 Layers)

Layer Responsibility
L1: Data Source Layer Identify data sources (DBs, files, web), formats, volume; push or pull mechanism.
L2: Data Ingestion Layer Real-time or batch ingest; ETL; preprocessing for storage/processing.
L3: Data Storage Layer Reliable, fault-tolerant, scalable storage. HDFS, HBase, Cassandra, MongoDB.
L4: Data Processing Layer Batch/real-time analysis. MapReduce, Spark, Hive, Pig, Mahout.
L5: Data Consumption Layer Reporting, dashboards, real-time analytics, decisions.

Processing Techniques

Term Definition
Batch Processing Process large volumes of data together at scheduled times. Focus = throughput, high latency. Tools: Hadoop, Spark.
Stream Processing Process continuous data immediately as produced. Focus = immediate response, low latency. Tools: Kafka, Spark Streaming, Flink.

Scaling

Term Definition
Vertical Scaling (Scale Up) Upgrade a single server's CPU/RAM/storage. Concentrates power. Expensive, hard limits.
Horizontal Scaling (Scale Out) Add multiple cheaper machines to share workload. "Divide and conquer." Used in Big Data.

Load Balancing

Term Definition
Load Balancer Distributes incoming requests/workloads across multiple servers; prevents single-machine overload.
Round Robin Requests distributed equally in order (Server1 → 2 → 3 → repeat).
Least Connections Routes to server with fewest active connections.
Resource-Based Routes to server with highest available CPU/RAM.
IP Hash Maps client IP to a fixed server for session persistence.

Cluster & Storage

Term Definition
Cluster Group of interconnected servers (nodes) working together as a single system.
Node One machine in the cluster (own CPU/RAM/storage; cooperates with others).
Tightly Coupled System Nodes connected with high-speed interconnects, minimal latency.
On-Disk Storage HDD/SSD; persistent, cheaper, slower. For batch + long-term storage.
In-Memory Storage Data in RAM; very fast, expensive, volatile. For real-time/streaming.
File System Method to store/organize data on storage devices — logical tree of files & directories.
Distributed File System (DFS) Stores large files across many nodes; appears local to users (logical view). E.g., GFS, HDFS.

Data Distribution Models

Term Definition
Replication Storing the same data on multiple nodes. Ensures availability & fault tolerance.
Sharding Horizontally partitioning a dataset; different rows on different nodes. Improves scalability.
Sharding + Replication Combination: dataset is sharded, and each shard is replicated. Common in production.

Replication Models

Term Definition
Master–Slave Replication Master handles all writes; slaves replicate from master and handle reads.
Master–Master (Peer-to-Peer) Multiple nodes can accept writes; replicate to each other. Higher write throughput but harder consistency.

L02 — HDFS

Core HDFS

Term Definition
HDFS Hadoop Distributed File System — distributed storage for massive files on commodity hardware.
Commodity Hardware Cheap, off-the-shelf machines (not enterprise-grade).
Streaming Data Access Optimized for high-throughput sequential reads, not low-latency random access.
Write-Once-Read-Many HDFS design assumption: files are written once, then read many times.

Blocks

Term Definition
Block Smallest physical storage unit in HDFS. Default size = 128 MB.
Block Abstraction Users see files (logical); HDFS internally splits files into fixed-size blocks distributed across DataNodes.
Block Size Trade-off Larger blocks → fewer tasks, possible idle nodes. Smaller blocks → many tasks, high scheduling overhead.
Sharding (vs Blocks) App-level logical partitioning. Blocks are physical storage units done by HDFS.

Architecture

Term Definition
Master-Slave Architecture HDFS pattern: ONE NameNode (master) + multiple DataNodes (workers).
NameNode Master daemon that manages metadata (directory structure, file→block mapping, permissions). Single point of failure in Hadoop 1.x.
DataNode Worker daemon that stores actual data blocks and handles all I/O operations.
FsImage (Filesystem Image) Snapshot of the entire filesystem namespace at a point in time. Contains directory structure, file→block mapping, replication factor, permissions. No actual block data.
Edit Log Records every metadata change since the last FsImage snapshot. Examples: file create/delete/rename, permission updates.
Snapshot + Transaction Log The two-file design (FsImage + Edit Log) — improves NameNode performance vs rewriting FsImage on every change.

Health Monitoring

Term Definition
Heartbeat Messages DataNode → NameNode "I'm alive" pings every few seconds.
Block Report Complete list of blocks the DataNode currently holds. Sent periodically (longer interval than heartbeat).
Under-replication Detection NameNode notices a block has fewer than RF replicas → initiates re-replication.

Secondary NameNode

Term Definition
Secondary NameNode Helper node that periodically merges FsImage + Edit log into a fresh FsImage. NOT a backup. NOT a failover. Only does checkpointing.
Checkpointing The merge operation that produces a new FsImage from old FsImage + edit log → keeps edit log small.

Small Files Problem

Term Definition
Small Files Problem Each file (regardless of size) consumes ≥ 1 block + 150 bytes of NameNode RAM. Millions of small files exhaust NameNode memory.
Metadata RAM NameNode stores all file/block metadata in RAM. ~150 bytes per block.

HDFS Balancer

Term Definition
HDFS Balancer Admin tool that evenly distributes data across DataNodes to prevent imbalance.
Threshold (T) Percentage threshold (e.g., 10%) above which a node is considered imbalanced.
Uᵢ (Node Usage) (Used space on node / Total capacity) × 100.
U_avg (Cluster Average Usage) (Σ used) / (Σ capacity) × 100.
Dᵢ (Deviation) |Uᵢ − U_avg|. If > T → node imbalanced (over or under-utilized).

Data Flow

Term Definition
Read Pipeline Client → NameNode (metadata) → directly contacts DataNodes for parallel block reads.
Write Pipeline Data flows DN1 → DN2 → DN3 (sequential pipeline) for replicas. Acks flow back DNn → DN1 → client.
Pipeline Recovery On DataNode failure mid-write: bad node removed, write continues on remaining; NameNode later re-replicates.
Acknowledgment (Ack) Confirmation sent back through the pipeline after a block replica is successfully written.

Fault Tolerance

Term Definition
Replication Factor (RF) How many copies of each block are stored. Default = 3.
Replication Overhead File size on disk = RF × original file size.
Rack Group of DataNodes connected to the same network switch. Faster intra-rack communication.
Rack Awareness Hadoop knows which nodes belong to which rack — used for intelligent replica placement.
Default Rack-Aware Placement (RF=3) 1st replica: local node. 2nd replica: different node in same rack. 3rd replica: different rack.

Erasure Coding (EC)

Term Definition
Erasure Coding (EC) Feature in Hadoop 3.0 — same fault tolerance as 3× replication, but with lower storage overhead.
EC Policy Defines how data and parity blocks are organized. Format: RS-K-M.
RS-K-M Reed-Solomon with K data cells + M parity cells. Tolerates up to M failures; physical storage = (K+M)/K × original.
Data Cells The K cells of original data.
Parity Cells The M extra cells computed mathematically from data cells; used to reconstruct lost data.
EC vs Replication EC slower writes & recovery (math), much less storage. Used for warm/cold data. Replication: fast writes/recovery, 3× storage. Used for hot data.

HDFS Federation & High Availability

Term Definition
HDFS Federation Multiple independent NameNodes in the same cluster — each manages part of the namespace. Solves scalability.
Block Pool Set of blocks belonging to a single namespace.
Namespace Volume Namespace + its block pool.
High Availability (HA) Active + Standby NameNode pair — if Active fails, Standby takes over. Solves SPOF.
Active NameNode The currently running NameNode handling all requests.
Standby NameNode Hot backup with identical state; takes over instantly on Active's failure.
Failover Process of switching from a failed Active to the Standby.

L03 — MapReduce

Parallel Programming Models

Term Definition
Parallel Programming Model Abstraction specifying how parallel tasks are created, coordinated, and executed.
Shared Memory Model Multiple threads/processes access common shared memory. One machine. E.g., OpenMP, pthreads.
Message Passing Model Each processor has private local memory; communication via explicit send/receive. Programmer manages comms. E.g., MPI.
Data Parallel Model Same operation applied concurrently on different data partitions. E.g., MapReduce.

MapReduce Concepts

Term Definition
MapReduce Google 2008 data-parallel programming model — hides parallelism, fault tolerance, data distribution from the programmer.
Map Function User-defined function that transforms input (k,v) → list of intermediate (k2,v2).
Reduce Function User-defined function that aggregates/summarizes all values for one key into final output.
Combiner ("Mini-Reducer") Optional optimization — partial merging of intermediate data on the mapper side, before shuffle. Works ONLY if reduce is associative + commutative (Sum, Count, Min, Max).
Partitioner Decides which reducer gets each key. Default: hash(key) % #reducers.
Data Skew When one key dominates → one reducer overloaded.
Key Salting Splitting hot keys into subkeys (e.g., "the_1, the_2") to distribute load.
Range Partitioning Assign keys based on range (R1: A-F, R2: G-M, …). Works if data is uniform/sorted.
Frequency-Aware Partitioning Distribute keys based on observed frequencies.

MapReduce Pipeline Components

Term Definition
Input Format Recipe telling Hadoop how to split file + read records.
Input Split Logical reference (pointer) to a portion of a file processed by one Mapper. Usually 1 split = 1 block. 1 split = 1 Map task.
Record Reader Converts a split's bytes into (key, value) records. For text: key = byte offset, value = line content.
Byte Offset Position in file (in bytes) where a record begins — unique per record.
Mapper Component that runs the Map function on a split, producing intermediate (k,v) pairs.
Intermediate Key-Value Pairs Output of Mappers, before Shuffle/Reduce. Keys can be duplicated.
Shuffle Phase Network transfer of intermediate (k,v) pairs from Mappers → Reducers. Groups records by key. Most expensive phase.
Sort Phase Sorting intermediate pairs by key before sending to reducer. Enables efficient aggregation.
Reducer Receives (key, [list of values]) and merges/aggregates into final output.
Record Writer Writes final output to HDFS. Each Reduce task produces one output file.

MRv1 Architecture (Old)

Term Definition
JobTracker Centralized master daemon in MRv1 — manages scheduling, resource allocation, monitoring, task failure handling. Single point of failure.
TaskTracker Worker daemon running on DataNodes — executes Map/Reduce tasks, reports heartbeats.
Slot Fixed bundle of resources to run a job in MRv1. Each node had fixed Map slots & Reduce slots — caused inefficiency.

Fault Tolerance & Performance

Term Definition
Task Re-Execution On failure, task is re-run on another node. Data safe in HDFS.
Speculative Execution If one task is unusually slow ("straggler"), system launches a duplicate copy; first to finish wins.
Straggler A task running much slower than peers (overloaded node, faulty disk).
Data Locality Run computation on the node holding the data — avoid network transfer.

When MapReduce is the Wrong Tool

Term Definition
Iterative Algorithms Algorithms that need many passes over same data (ML training, PageRank). MR rereads disk each iteration → slow.
Real-time / Low-Latency MR is high-latency batch — can't respond in ms.
Interactive Queries MR jobs take minutes → not interactive.
Sequential / Dependent Tasks Tasks where step N depends on N-1 don't fit Map+Reduce shape.

L04/L05 — YARN

YARN Core

Term Definition
YARN Yet Another Resource Negotiator — separates resource management from processing, introduced in Hadoop 2.0.
Multi-tenancy Multiple processing engines (MR, Spark, Flink, Tez) coexist on same cluster.
Daemon A background process — RM, NM are daemons.

Components

Term Definition
Client Job submitter — initiates jobs to RM, monitors progress.
Resource Manager (RM) Master daemon — global resource scheduling, container allocation, manages cluster capacity. "The cluster boss."
Node Manager (NM) Per-node daemon — manages containers and monitors resources on its node. "Building supervisor."
Application Master (AM) Per-job daemon — runs inside a container on a worker node, negotiates resources for its app, schedules tasks. NOT same as Application Manager!
Container Logical bundle of resources (RAM + CPU + disk) on a single node — where tasks run. Not a physical machine.

RM Sub-components

Term Definition
Scheduler Pure scheduler — allocates resources subject to capacity & queues. No monitoring.
Application Manager Sub-component of RM that accepts job submissions and starts the AM. Different from Application Master!

Resource Request & CLC

Term Definition
Resource Request AM → RM request with 3 fields: Resource Name (host/rack/anywhere), Priority, Capability (CPU/RAM).
Container Launch Context (CLC) Instructions sent from AM → NM telling how to start a container: env vars, app dependencies (code/libs), security tokens.
Security Tokens Issued by RM to authorize AM to launch containers safely.

Schedulers

Term Definition
FIFO Scheduler First-in, first-out queue. Simple. Can cause starvation. Suitable only for single-user test clusters.
Fair Scheduler Every running app gets an equal share over time. Dynamic rebalancing. No starvation. Fairness by memory (configurable for CPU).
Capacity Scheduler Hierarchical queues; each org/team gets a guaranteed fraction of cluster. Production standard.
Queue A bucket in the scheduler holding submitted apps.
Root Queue Top of capacity scheduler hierarchy — 100% cluster capacity.
Parent Queue Organizational container. Does NOT accept jobs.
Leaf Queue Bottom of hierarchy. Only leaf queues accept job submissions.
Guaranteed Capacity Minimum % of cluster dedicated to a queue.
Resource Elasticity A queue can borrow idle capacity from other queues.
Starvation A small job waits forever because a large job hogs all resources (FIFO problem).

High Availability & Failure Handling

Term Definition
ZooKeeper External coordination service — manages distributed systems' "who's the leader?"
Leader Election (ZK role) Ensures only ONE Active RM at a time; promotes Standby on failure.
State Storage (ZK role) Stores RM's app state/metadata so new Active RM can recover running apps without data loss.
Active / Standby Pair RM HA pattern — one Active, one Standby ready to take over.
Blacklisted Node A node that stopped sending heartbeats → RM marks it as bad → containers rescheduled elsewhere.

Failure Recovery (memorize the table)

Failure Recovery
Container fails AM retries (default 4 attempts)
Node Manager fails RM blacklists node, containers rescheduled
Application Master fails RM restarts AM in a new container
Resource Manager fails ZooKeeper triggers failover to Standby RM

Data Locality in YARN

Term Definition
Node-local Container on the exact DataNode holding the block (best).
Rack-local Container on same rack, different node (1 hop).
Off-rack Different rack (multi-hop, slowest).
Scheduling Delay RM intentionally waits a few seconds for a node-local container before falling back to rack-local.

L07 — Spark Core / RDD

Spark Basics

Term Definition
Apache Spark Open-source, in-memory data processing engine. Up to 100× faster in memory, 10× on disk vs MapReduce.
In-Memory Computing Keep intermediate results in RAM, avoiding disk I/O. Spark's killer feature.
Spark Ecosystem Spark Core + Spark SQL + Spark Streaming + MLlib + GraphX.
Spark Core Foundation — task scheduling, basic I/O, the RDD API.

Spark's 2 Core Abstractions

Term Definition
RDD (Resilient Distributed Dataset) Fundamental data abstraction — distributed collection of elements processable in parallel. Not a single contiguous file; divided into partitions.
DAG (Directed Acyclic Graph) Graph of dependencies between RDDs — represents the workflow Spark optimizes.

RDD 3 Pillars

Term Definition
Resilient Fault tolerance via lineage. Lost partitions are recomputed using their lineage.
Distributed Automatically split into partitions across cluster nodes → parallelism.
Datasets Generic container for any object type (Scala/Java/Python).

RDD Properties

Term Definition
Immutability Once created, an RDD cannot be changed. Avoids race conditions.
Race Condition Bug where multiple tasks try to update the same data simultaneously → corruption.
Lineage Graph Recorded chain of transformations from base RDD to current RDD. Used for re-computation on failure.
Partition Subset of an RDD's data on one node. Unit of parallelism.

RDD Operations

Term Definition
Transformation Creates a new RDD from an existing one. Lazy — doesn't execute immediately. Returns a NEW RDD.
Action Forces execution and returns a value or writes to storage. Eager.
Narrow Transformation Each input partition contributes to exactly ONE output partition. No shuffle. E.g., map, filter, flatMap.
Wide Transformation Input partitions contribute to MULTIPLE output partitions → shuffle across cluster. E.g., reduceByKey, groupByKey, join.
Shuffle Boundary A wide transformation = stage break in the DAG.
Common Transformations map, filter, flatMap, reduceByKey, groupByKey, join, union
Common Actions collect, count, reduce, saveAsTextFile, take, foreach
persist() / cache() Save an RDD in memory (or disk) so future actions don't re-execute the lineage.

Lazy Execution

Term Definition
Lazy Execution Transformations don't run until an action triggers execution. Spark builds a logical plan first.
Benefits of Lazy 1) Whole-plan optimization. 2) Skip unused computations. 3) Easier fault tolerance via lineage.

DAG Execution Model

Term Definition
Job Unit of work triggered by one action.
Stage Group of narrow transformations that run together; boundaries are wide transformations (shuffles).
Task One unit of work on one partition. Tasks within a stage run in parallel on executors.
Logical Plan High-level lineage graph derived from user code.
Physical Plan Actual execution graph of stages + tasks, after optimization.

Spark Architecture

Term Definition
Driver Program "Brain" — runs the main app, creates DAG, divides work into stages/tasks, schedules execution.
Cluster Manager Allocates CPU/RAM. E.g., YARN, Mesos, Kubernetes, Standalone.
Worker Node Machine in the cluster that hosts executors.
Executor Worker process that runs tasks, stores cached RDD data in memory, handles shuffle, returns results to driver.

Optimization Techniques

Term Definition
Lazy Evaluation Build optimized plan, defer execution.
Data Locality (Spark) Schedule tasks where data lives.
In-Memory Computing Keep intermediates in RAM.
Speculative Execution (Spark) Duplicate stragglers, fastest wins.

L08 — Spark SQL & DataFrames

Term Definition
Structured APIs DataFrames + Datasets — introduced in Spark 1.3 to simplify analytics with automatic optimization.
DataFrame Structured, distributed dataset with schema — like a relational table or pandas DataFrame.
Dataset Typed version of DataFrame (Scala/Java).
Schema Named columns + their data types.
Project Tungsten Custom memory management for DataFrames — reduces RAM usage & garbage collection overhead.
Catalyst Optimizer Query optimizer for DataFrames/SQL — produces efficient execution plans automatically.
Rule-Based Optimization (RBO) Apply deterministic rewrite rules (e.g., push filters before joins).
Cost-Based Optimization (CBO) Choose the cheapest plan using statistics (row counts, data sizes).
Predicate Pushdown Move filtering as close to the data source as possible — only read rows that match.
Column Pruning Read only the columns you need from the source; skip irrelevant ones.
Constant Folding Pre-compute constant expressions (e.g., 1+2 → 3) at compile time.
Join Strategy Choose the best join method based on data size: Broadcast / Shuffle Hash / Sort Merge.
Broadcast Join Small table sent ("broadcast") to all nodes — fast when one side fits memory.
Shuffle Hash Join Partition-based join — hash-shuffle both sides on join key.
Sort Merge Join For large, sorted datasets — sort both sides, then merge.
Query Analysis Catalyst stage 1: resolve tables/columns.
Logical Optimization Catalyst stage 2: apply Predicate Pushdown, Column Pruning, Constant Folding.
Physical Planning Catalyst stage 3: generate multiple physical plans, choose cheapest (cost-based).
Code Generation Catalyst stage 4: compile plan to JVM bytecode (Tungsten).
Spark SQL Module Module that lets you query DataFrames using SQL via spark.sql().
DAG Scheduler (Spark) Splits the DAG into stages at shuffle boundaries; pipelines narrow ops; generates tasks.
Task Scheduler Assigns tasks to executors.
Stage Pipelining Combine narrow operations (map + filter) within a single stage to avoid RAM/disk round-trips.
DataFrame Operations select, filter / where, join, orderBy / sort, distinct, groupBy, union, show(n), collect, count, take(n) / head(n).

L09 — Spark ML & GraphX

MLlib Basics

Term Definition
MLlib Spark's distributed Machine Learning library.
spark.mllib Older MLlib package — RDD-based.
spark.ml Modern MLlib package — DataFrame-based.
Transformer Component that converts one DataFrame → another. Implements .transform(). E.g., feature scalers, trained models.
Estimator Learning algorithm. Implements .fit(data) → Transformer. The output Transformer is the trained model.
Pipeline Sequence of stages (Transformers and/or Estimators) run in order.
PipelineModel Output of Pipeline.fit() — pipeline where every Estimator has been replaced by its trained Transformer.

Pipeline Stages

# Stage What it does
1 Data Input Load data into DataFrame (CSV, Parquet, Hive…).
2 Feature Engineering Convert raw data → numerical features.
3 Model Training Fit a model (Logistic Regression, Decision Tree…).
4 Evaluation Measure accuracy / precision / recall.
5 Prediction Apply trained model to new data.

Distributed Training Workflow

Term Definition
Broadcast Driver sends model parameters to all workers.
Partial Results Each worker computes gradients (for regression) or splits (for trees) on its partition.
Aggregation Workers send partial results back; driver combines them.
Model Update Global model updated with aggregated info.

Optimization Algorithms

Term Definition
Gradient Descent (GD) Iterative optimization that uses the entire local partition per worker per iteration.
Stochastic Gradient Descent (SGD) Variant where each worker samples a small subset of its partition. Faster per iteration, noisier.
treeAggregate Spark's efficient hierarchical aggregation operation — used to reduce gradients across workers.

Decision Trees

Term Definition
Decision Tree Tree-based predictor — at each node, splits data by feature value.
Gini Impurity Measure of node "impurity": 1 − Σ pᵢ². Lower = purer.
Entropy Alternative impurity measure: −Σ pᵢ log₂ pᵢ.
Information Gain Reduction in impurity from a split (parent impurity − weighted child impurity).
Best Split The feature + threshold that maximizes information gain.

Graph Theory

Term Definition
Graph Data structure representing relationships between entities.
Vertex (Node) Real-world entity (user, product, page).
Edge (Link) Relationship between two vertices (friendship, purchase, hyperlink).
Directed Multigraph Graph where edges have direction AND multiple edges between same vertex pair are allowed.

GraphX

Term Definition
GraphX Spark's component for graph processing. Extends RDD with a Graph abstraction.
Property Graph Directed multigraph with attributes on vertices and edges.
VertexId Unique 64-bit ID for each vertex.
VertexRDD[VD] Optimized vertex collection: (VertexID, Attribute).
EdgeRDD[ED] Optimized edge collection: (SourceID, DestinationID, Attribute).
Graph Triplet (srcId, srcAttr, dstId, dstAttr, edgeAttr) — joined view of edge + endpoint vertices. Used by algorithms that need both.

Graph Operators (4 families)

Family Examples
Property Operators mapVertices, mapEdges, mapTriplets
Structural Operators reverse, subgraph, groupEdges
Join Operators joinVertices, outerJoinVertices
Neighborhood Aggregation aggregateMessages

Graph Partitioning

Term Definition
Edge-Cut Partitioning (vertex-based) Each vertex assigned uniquely to one machine; edges may cross machines. Bad for power-law graphs (hub hotspots).
Vertex-Cut Partitioning (edge-based) Each edge assigned to one partition; vertices replicated. GraphX uses this — balances load.
Master / Mirror In Vertex-Cut: one machine holds the master copy of a vertex; others hold mirrors.
VertexTable GraphX RDD: (id, data) — vertex attributes.
EdgeTable GraphX RDD: (pid, src, dst, data) — edges by partition.
Routing Table GraphX RDD: (id, pid) — tells where vertex mirrors are; enables efficient joins.
Power-Law Graph Real-world graph distribution where few vertices have huge degree (hubs) and most have small degree. Social networks, web.

PageRank

Term Definition
PageRank Algorithm that scores a vertex's importance based on incoming links from important vertices.
Random Surfer Mental model: user randomly clicks links (prob d) or jumps to random page (prob 1−d).
Damping Factor (d) Probability of following a link (typically 0.85).
Outdegree Number of outgoing edges of a vertex.
PageRank formula PR(v) = (1−d) + d × Σ PR(u)/outdeg(u) for all u → v.
Convergence Stop iterating when rank change < tolerance OR max iterations reached.
PageRank Algorithm (Distributed) 4 phases per iteration: Map (PR(u)/outdeg(u)) → Shuffle → Reduce (aggregate) → Broadcast → repeat.

L10 — Spark Streaming & Kafka

Streaming Basics

Term Definition
Data Streaming Real-time processing of continuous, unbounded data in motion.
Batch vs Stream Batch = throughput-focused, high latency. Stream = low-latency, immediate response.
Micro-batch Processing Process data in small, fixed-size batches (Spark Streaming approach).
Real-time Streaming Process events as they arrive, no batching (Flink).
Streaming Pipeline Sources → Ingestion (Kafka) → Processing (Spark) → Serving (dashboards/alerts).

Apache Kafka

Term Definition
Apache Kafka Open-source distributed event-streaming platform. Built by LinkedIn. Moves data efficiently — does not analyze.
Producer Application that sends (publishes) messages to Kafka brokers.
Consumer Application that reads (consumes) messages from Kafka topics.
Topic Logical channel where producers publish and consumers read. Messages are immutable once published.
Broker Server that stores and manages streaming data. Handles read/write.
Cluster (Kafka) Group of brokers working together.
Kafka ZooKeeper Coordinates brokers, tracks availability, notifies producers/consumers when brokers fail.
Partition Sub-division of a topic; ordered, append-only log. Enables parallel consumption and scaling.
Leader (partition role) Handles all read & write operations for that partition.
Replica (partition role) Maintains a copy of the leader's data; syncs continuously; promoted to leader on failure.
Data Log Kafka stores messages in a log structure for a configurable retention period.
Retention Period How long messages are kept in Kafka. If consumer is offline longer → messages may be lost.

Spark Streaming

Term Definition
Spark Streaming Extension of Spark for real-time stream processing.
DStream (Discretized Stream) Spark 1.x streaming model — continuous sequence of small RDDs (micro-batches).
Structured Streaming Spark 2.x+ model — treats live stream as an append-only table; uses DataFrame API + SQL engine.
Input Table In Structured Streaming, the logical table that grows as new data arrives.
Result Table In Structured Streaming, the incrementally updated query result.
Output Sink Destination of processed results (database, Kafka, console, file…).

Output Modes

Mode What it writes
Complete Mode Entire Result Table every trigger. For aggregations.
Append Mode Only new rows since last trigger. For pure appends.
Update Mode Only changed rows since last trigger. Efficient incremental.

Transformations

Term Definition
Stateless Transformation Each batch processed independently — no memory of past. E.g., map, filter, flatMap.
Stateful Transformation Uses both current + past batches. Maintains state across time. E.g., updateStateByKey, windowed aggs.
Windowed Operations Aggregations over a sliding/tumbling time window (e.g., count events in last 30 sec).
updateStateByKey DStream method that maintains state per key across all batches.

Event-Time & Late Data

Term Definition
Event Time The timestamp embedded in the event (when it actually happened).
Processing Time When Spark received/processed the event.
Late Data Events that arrive after their event-time window has been processed.
Watermark Threshold defining how late an event can arrive and still be processed. Events beyond watermark → dropped.
Watermarking Mechanism for bounded-memory event-time aggregation. Defines how to handle late data.

Window Types

Type Definition
Tumbling Window Fixed-size, NON-overlapping, contiguous intervals. Each event in exactly one window.
Sliding Window Fixed-size, OVERLAPPING (slide < window size). Each event in multiple windows.
Session Window Dynamic size — starts with an event, expands while next event arrives within gap.

Checkpointing

Term Definition
Checkpointing Save app state to reliable storage (HDFS) for fault tolerance in long-running streaming jobs.
Metadata Checkpointing Saves config, DAG, queue of unprocessed batches.
Data Checkpointing Saves actual RDD data for stateful transformations.

Other Streaming Tools

Term Definition
Apache Flink True real-time stream processor (not micro-batch). Also supports batch.
Apache Storm Real-time stream processor (older).
Kafka Streams Streaming library inside Kafka — process streams without separate cluster.

L11 — Data Warehouses, Data Lakes, Lakehouses

Term Definition
Data Warehouse (DW) Stores structured data with schema-on-write. Optimized for SQL queries and BI. High reliability, expensive, rigid.
Data Lake (DL) Stores raw data of all types (structured/semi/unstructured) with schema-on-read. Cheap, flexible, prone to "data swamp."
Data Lakehouse Hybrid — combines Lake's flexibility + Warehouse's reliability. Uses object storage + Table Layer for ACID/structure.
Table Layer Software layer on top of Data Lake that organizes raw files into reliable analytical tables. Provides DB-like semantics.
Transaction Log Tracks every change for ACID guarantees (Table Layer feature).
Schema Evolution Add columns without rewriting all existing files.
Time Travel Query historical versions of the data.
Partition Pruning Query optimization — skip files that don't match filter (Table Layer feature).
ACID Transactions Atomicity, Consistency, Isolation, Durability — DB guarantees added by Table Layer.
Data Swamp Pejorative for an unmanaged Data Lake that lost quality and discoverability.
Object Storage Flat-file storage system (S3, ADLS, GCS, HDFS) — basis for Lakes & Lakehouses.
Parquet Columnar file format optimized for analytics.

Cross-Cutting Terms

These appear across multiple lectures.

Term Definition
Single Point of Failure (SPOF) A component whose failure brings down the entire system. NameNode (1.x), JobTracker (MRv1), RM (pre-2.4).
Heartbeat Periodic "I'm alive" message between worker and master. Used by HDFS, MRv1, YARN.
Data Locality Run computation where data lives — avoids network transfer. Used by HDFS-aware MapReduce, YARN, Spark.
Master-Slave Architecture One coordinator + many workers. HDFS, MRv1, YARN, Spark all follow this pattern.
Fault Tolerance Continued operation despite failures. HDFS uses replication; Spark uses lineage; YARN uses HA + restarts.
Throughput Volume of work per unit time. Optimized by batch systems.
Latency Delay between request and response. Optimized by streaming systems.
Shuffle Network movement of intermediate data — most expensive phase in distributed compute (MapReduce + Spark).
Lazy Evaluation Defer computation until needed. Used in Spark transformations.
Speculative Execution Run duplicate of slow task; fastest wins. Used in MapReduce + Spark.
Container (YARN) Resource bundle running tasks. Replaced MRv1's rigid slots.
Slot (MRv1) Fixed resource bundle, deprecated.
Schema (DW/DL/Spark) Definition of data structure — column names + types.
Partition (HDFS / Spark / Kafka) Subdivision of data for parallelism. Different meaning per system: in HDFS = block, in Spark/Kafka = logical chunk.
Replica / Replication (HDFS + Kafka) Copy of data for fault tolerance.
ZooKeeper (YARN + Kafka) External coordinator for leader election and shared state.

📋 Master Numerical Cheat-Card (one screen)

HDFS:
  default block        = 128 MB
  default RF           = 3
  metadata per block   = 150 bytes
  blocks/file          = ⌈ file_size / block_size ⌉
  metadata RAM         = total_blocks × 150 B
  physical storage     = blocks × block_size × RF

BALANCER:
  Uᵢ                   = used_i / cap_i × 100
  U_avg                = Σ used / Σ cap × 100
  Dᵢ                   = |Uᵢ − U_avg|
  imbalanced if        Dᵢ > T

ERASURE CODING (RS-K-M):
  K data + M parity    = K+M cells stored
  overhead             = (K+M)/K
  tolerates            = M failures
  e.g., RS-6-3: 50% saved vs 3× rep; tolerates 3 failures
  e.g., RS-10-4: ~53% saved; tolerates 4 failures

YARN:
  max nodes            ≈ 10,000+ (MRv1 ≈ 4000)

SPARK:
  speed                = 100× in memory, 10× on disk vs MR
  partition  per       = ~one task

PAGERANK:
  contribution(u→v)    = PR(u) / outdegree(u)
  PR(v)                = (1−d) + d × Σ contributions
